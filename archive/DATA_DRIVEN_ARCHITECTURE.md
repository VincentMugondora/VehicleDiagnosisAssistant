# Data-Driven Architecture Refactoring

## Summary

Refactored the Vehicle Diagnosis Assistant to eliminate heuristic logic and move all diagnostic intelligence from the formatter layer to the data layer. The formatter is now presentation-only, with all automotive knowledge stored in the database or generated by AI.

## Architecture Before

```
┌──────────────────────────────────────────────┐
│          OBD Service                         │
│  ├─ Database lookup                          │
│  ├─ Returns: code, description, causes       │
│  └─ Missing: symptoms, severity, tips        │
└──────────────────────────────────────────────┘
                    ↓
┌──────────────────────────────────────────────┐
│     Diagnostic Formatter (446 lines)         │
│  ├─ KEYWORD MATCHING on description          │
│  ├─ Generate symptoms from "sensor"/"misfire"│
│  ├─ Infer severity from "knock"/"catalyst"   │
│  ├─ Generate tips based on code patterns     │
│  └─ 🔴 Contains automotive knowledge         │
└──────────────────────────────────────────────┘
                    ↓
              WhatsApp User
```

**Problems:**
- Formatter contained 300+ lines of automotive heuristics
- Keyword matching unreliable (`"knock"` in different contexts)
- Hard to maintain (knowledge scattered across codebase)
- No way to improve without editing code
- Same code analyzed differently based on keywords

## Architecture After

```
┌──────────────────────────────────────────────┐
│         Database + AI Layer                  │
│  ├─ Stores ALL fields for each code          │
│  ├─ symptoms, severity, technician_tip       │
│  ├─ pre_replacement_checks, explanation      │
│  └─ AI enriches missing fields automatically │
└──────────────────────────────────────────────┘
                    ↓
┌──────────────────────────────────────────────┐
│          OBD Service                         │
│  ├─ Checks database completeness             │
│  ├─ Triggers AI enrichment if needed         │
│  ├─ Saves enriched data back to database     │
│  └─ Returns fully populated DiagnosticResult │
└──────────────────────────────────────────────┘
                    ↓
┌──────────────────────────────────────────────┐
│   Diagnostic Formatter (202 lines)           │
│  ├─ PURE PRESENTATION                        │
│  ├─ No keyword matching                      │
│  ├─ No automotive knowledge                  │
│  └─ ✅ Just formats DiagnosticResult         │
└──────────────────────────────────────────────┘
                    ↓
              WhatsApp User
```

**Benefits:**
- Formatter simplified from 446 to 202 lines (54% reduction)
- Removed all keyword matching and heuristic inference
- All automotive knowledge in database (queryable, auditable)
- AI improves data quality over time
- Easy to correct: just update database, not code

## Changes Made

### 1. Expanded DiagnosticResult Model

**File:** `app/models/diagnostic.py`

**Added fields:**
```python
severity: str | None = None  # "Critical" | "High" | "Moderate" | "Low"
severity_explanation: str | None = None  # Why this severity level
technician_tip: str | None = None  # Practical diagnostic tip
pre_replacement_checks: list[str] | None = None  # What to verify first
```

**Why:** These fields were previously inferred by the formatter using keyword matching. Now they come from the database or AI generation.

### 2. Enhanced AI Code Generator

**File:** `app/services/ai_code_generator.py`

**Changes:**
- Updated prompt to generate ALL enrichment fields
- Added validation for new fields (severity, technician_tip, pre_replacement_checks)
- Improved severity mapping (Critical/High/Moderate/Low)
- Made generation more robust with fallbacks

**Why:** AI now generates complete diagnostic data, not just basic fields.

### 3. Added Intelligence to OBD Service

**File:** `app/services/obd_service.py`

**New functionality:**
- **Enrichment Detection:** Checks if database entry has all required fields
- **Auto-Enrichment:** Calls AI to fill missing fields when needed
- **Database Update:** Saves enriched data back to database for future use
- **Removed Heuristics:** Deleted 144 lines of keyword-matching methods:
  - `_generate_generic_causes()` (removed)
  - `_generate_generic_checks()` (removed)

**Key method added:**
```python
async def _enrich_and_save(self, code: str, base_data: dict) -> Optional[dict]:
    """
    Enrich existing database entry with missing fields using AI.
    Saves enriched data back to database.
    """
```

**Why:** Intelligence moved from formatter to data layer. Service ensures all results are fully populated.

### 4. Simplified Formatter to Presentation-Only

**File:** `app/services/diagnostic_formatter.py`

**Before:** 446 lines with automotive knowledge
**After:** 202 lines, pure presentation

**Removed:**
- `_generate_common_symptoms()` - 115 lines of keyword matching
- `_determine_severity()` - 60 lines of heuristic inference
- `_generate_technician_tip()` - 80 lines of pattern matching
- `_generate_pre_replacement_checks()` - 80 lines of conditional logic

**Kept:**
- Section formatting methods (header, description, lists)
- Single minimal fallback for severity explanation
- All presentation logic intact

**Why:** Formatter should only present data, not create it. Separation of concerns.

### 5. Updated Tests

**New file:** `tests/test_data_driven_formatter.py`

**Tests validate:**
- Formatter is presentation-only (no automotive knowledge)
- No keyword matching occurs
- All intelligence comes from DiagnosticResult
- Graceful handling of missing fields
- Consistency across data sources

**Why:** Ensure architectural principles are maintained.

## Key Principles

### 1. Separation of Concerns

| Layer | Responsibility | Contains |
|-------|---------------|----------|
| **Database** | Store diagnostic knowledge | Facts, symptoms, severity, tips |
| **AI Generator** | Create diagnostic knowledge | LLM-generated enrichment data |
| **OBD Service** | Orchestrate data flow | Lookup, enrichment, caching |
| **Formatter** | Present data | Markdown formatting, section layout |

### 2. No Heuristic Inference

❌ **Before:**
```python
if "knock sensor" in description:
    tip = "Check wiring first, not sensor"
```

✅ **After:**
```python
# technician_tip comes from database or AI
return self.result.technician_tip
```

### 3. Data Quality Improves Over Time

- **First lookup:** AI generates data
- **Saved to database:** Future lookups instant
- **Manual corrections:** Just update database
- **No code changes:** Improvements don't require deployment

### 4. Testability

**Before:** Hard to test keyword matching
```python
# How do we test all keyword combinations?
if "sensor" in desc and "circuit" not in desc:
    ...
elif "sensor" in desc and "circuit" in desc:
    ...
```

**After:** Easy to test presentation
```python
# Test that formatter presents provided data
assert report.contains(result.technician_tip)
```
